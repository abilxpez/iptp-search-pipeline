# IPTP Search Pipeline: Design Decisions

## Goals
- Support ~2,000 policy entries, each with multiple PDFs and multi-page PDFs.
- Free/open-source only (no paid APIs).
- Start with a small manual dataset (e.g., 5 entries) and scale without rewriting.

---

## Data Model: Entry vs Attachment vs Page vs Chunk

### Entry
An **entry** is the user-facing IPTP record (title, summary, filters, commentary, etc.).
- Stored in: `data/manual_entries.json`
- One JSON object per IPTP entry.

### Attachment
An **attachment** is a PDF connected to an entry.
- Example: one policy entry may include 1–3 PDFs.
- Each PDF is assigned an internal `attachment_id` (e.g., `pdf_1`, `pdf_2`) and a file path.

### Page
A **page** is one page of a PDF after text extraction.
- Pages preserve page numbers for citations and snippet display.

### Chunk
A **chunk** is a smaller passage used for retrieval and ranking.
- Chunks improve search quality vs indexing whole pages or whole PDFs.
- Each chunk retains `page_start` / `page_end` so results can cite pages.

---

## Why JSON for `manual_entries.json`
`manual_entries.json` is small and edited by humans.
- JSON (a single list) is easiest to manually view and edit.
- Contains metadata + attachment list + commentary links.

File:
- `data/manual_entries.json`

---

## Why JSONL for extracted/processed corpora
JSONL = “JSON Lines”: one JSON object per line.

### Key rule
- **Each line is an independent record**
- Enables streaming and incremental processing

This matters for scalability and reliability:
- We can process large corpora without loading everything into memory.
- If one PDF fails, we still keep all successfully processed records.
- Append-friendly: new documents can be added without rewriting old records.
- JSONL files are intermediate pipeline artifacts and are not queried directly at runtime.

---

## `extracted_pages.jsonl` (one line per PDF page)
This file is generated by `scripts/extract_pages.py`.

Each line represents a single PDF page:
- `entry_id`
- `attachment_id`
- `page_num` (1-indexed)
- `text` (extracted text for that page)
- `source_path` (local PDF path)

Why one line per page?
- Preserves page boundaries for accurate citations.
- Makes later chunking deterministic and traceable.

File:
- `data/extracted/extracted_pages.jsonl`

---

## `chunks.jsonl` (one line per chunk passage)
This file is generated by `scripts/chunk_pages.py`.

Each line represents a single chunk:
- `chunk_id` (stable identifier)
- `entry_id`
- `attachment_id`
- `page_start`, `page_end`
- `text` (chunk text)
- copied metadata needed for filtering/ranking (e.g., agency, subject_matter, dates)

Why chunking?
- Search works better when matching queries to smaller passages.
- Enables better snippets and relevance signals.

File:
- `data/chunks/chunks.jsonl`

---

## Index folders (generated artifacts)
Indexes are created by scripts and are not manually edited.

### BM25 index
- Stored under: `data/indexes/bm25/`
- Built from chunks + metadata (lexical retrieval)

### Vector index (FAISS)
- Stored under: `data/indexes/faiss/`
- Built from embeddings of chunks (semantic retrieval)

These folders are outputs of the pipeline and can be rebuilt as needed.
The pipeline supports incremental updates: new or changed PDFs can be reprocessed and reindexed without rebuilding the entire corpus.

---

## Scripts folder
All processing steps live in `scripts/`:
- `extract_pages.py`: PDF -> page-level JSONL
- `chunk_pages.py`: pages -> chunk-level JSONL
- `build_bm25.py` / `search_bm25.py`: lexical search MVP
- `embed_chunks.py` / `build_faiss.py`: semantic search
- `search_hybrid.py`: BM25 + vector fusion (RRF), optional reranking later

## Summary
- Human-authored metadata: `manual_entries.json` (JSON)
- Machine-generated corpora: `*.jsonl` (streaming, robust)
- Search happens on indexes (BM25 / FAISS), not on JSONL files.
- IDs (`entry_id`, `attachment_id`, `page_num`, `chunk_id`) ensure multi-PDF entries and citations work reliably.
